System Architecture

One process. Three logical stages. No network hop, no queue broker, no per-message disk write sits between message arrival and alert emission.

Generator Thread
Synthetic AIS-style position stream. Writes raw coordinate data into a shared, zero-copy ring buffer. Isolated so generation load never starves evaluation.
Hot Path — Main Thread
latLngToCell() → Set.has() → counter++ → violation branch. The entire per-message cost lives here, and it is O(1).
Alert / Metrics Layer
Violations pushed to WebSocket immediately. Throughput + latency sampled and broadcast every 200ms — never per message.
↓
Frontend Console
React application: live map, rolling throughput/latency charts, violation feed, dynamic zone overlay.

Design rationale

The dominant failure mode in comparable systems is treating this as a volume problem and reaching for distributed infrastructure (message brokers, stream processors, multi-service pipelines). It is not a volume problem — it is a lookup problem. The expensive part — determining whether a coordinate falls inside a protected polygon — is fully solvable once, offline, at startup, by pre-tiling every protected zone into H3 hexagon cells. What remains on the hot path is a single hash-set membership check per message, which does not get slower as zone geometry grows more complex and does not benefit from distribution.

Core Engine Design

Startup-time cost is deliberately front-loaded so that runtime cost per message is constant, regardless of how many zones exist or how complex their true geometry is.

Startup phase (once, not per-message)

// executed once at process start
const forbiddenSet = new Set();
for (const zone of protectedZonePolygons) {
  const cells = h3.polygonToCells(zone.coordinates, RESOLUTION);
  cells.forEach(cellId => forbiddenSet.add(cellId));
}

Hot path (once per incoming message)

// executed up to 50,000+ times per second
function evaluate(shipId, lat, lng, ts) {
  const t0 = process.hrtime.bigint();
  const cellId = h3.latLngToCell(lat, lng, RESOLUTION);
  messagesProcessed++;

  if (forbiddenSet.has(cellId)) {
    const t1 = process.hrtime.bigint();
    violationQueue.push({ shipId, cellId, ts, latencyNs: t1 - t0 });
  }
}

No allocation of new objects on the non-violation path beyond the primitive cell ID string; no array retention of processed (non-violating) messages; no synchronous I/O of any kind inside evaluate().

Data Flow & Message Schema

Inbound position message (binary frame, generator → evaluator)

[ shipId : 4 bytes ][ lat : 4 bytes ][ lng : 4 bytes ][ ts : 8 bytes ]  =  20 bytes / message

Fixed-width binary framing is used over the shared ring buffer specifically to avoid JSON parse cost at 50,000 messages/sec — a measurable and avoidable expense at this volume.

Concurrency Model

Deliberately minimal. Splitting evaluation logic across a worker pool was considered and rejected — the lookup is already O(1) and single-core throughput comfortably clears the floor, so added parallelism would introduce coordination cost without a corresponding benefit.

Thread allocation
ThreadResponsibilityIsolation reason
MainHot-path evaluation, metrics sampling, alert dispatchMust never be blocked by data generation
GeneratorSynthetic message production into SharedArrayBuffer ringPrevents generation bursts from starving the evaluator

If a worker pool is introduced later for a specific reason, the transport between threads must remain a SharedArrayBuffer / Atomics ring buffer — plain-object postMessage triggers structured cloning per call, which becomes the bottleneck it was meant to avoid.

Throughput measurement methodology

  • Rolling 1–2 second average displayed continuously — not an instantaneous peak, so a lucky burst cannot be mistaken for sustained capacity.
  • Load sustained for multiple continuous minutes during validation to surface memory growth or GC-pressure degradation before the live demo.
  • Metrics and violation broadcasts are batched/interval-based specifically so the observability layer cannot itself become the bottleneck it's meant to report on.

Frontend Technical Design

  • Map layer: GPU-accelerated rendering (Mapbox GL / deck.gl ScatterplotLayer / HexagonLayer) — never individual DOM markers, which collapse under thousands of points.
  • State updates: React state throttled to the same 200ms cadence as the metrics broadcast, not re-rendered per incoming message.
  • Rendered vessel count: a sampled subset is drawn for visual legibility; the evaluation engine — not the pixel count — is what is being measured against the throughput floor.
  • Metrics panel: two primary live numbers (messages/sec, latency ms) plus a rolling 30–60s line chart — the single highest-value proof surface for the constraint set.

Extended Feature Specifications

Each feature below extends detection coverage while remaining architecturally compatible with the throughput and latency constraints — every hot-path addition is O(1), and every periodic process runs isolated from message ingestion.

Dynamic Geofences — Whale Pod / Drift Zones

core hot-path compatible
Data structure: mutable Set<cellId>  ·  Update trigger: interval timer (5–10s)  ·  Hot path change: none

A protected zone's center follows a predefined waypoint path. On each movement tick, the system computes the new hexagon coverage for the updated center + radius, adds the new cells to the live Set, and removes cells no longer in range.

// runs on a timer, isolated from the ingestion loop
function tickZoneMovement(zone) {
  const newCenter = zone.nextWaypoint();
  const newCells  = h3.gridDisk(h3.latLngToCell(newCenter, RES), RADIUS_K);

  newCells.forEach(c => forbiddenSet.add(c));
  zone.currentCells.forEach(c => {
    if (!newCells.includes(c)) forbiddenSet.delete(c);
  });
  zone.currentCells = newCells;
}

An attached speed threshold is evaluated only against a vessel's hexagon membership in the zone's current cell set, so the rule automatically tracks the zone's live position with no additional state to reconcile.

Dark Fleet Detection — Transponder Drop-off

off hot-path sweep
Data structure: Map<shipId, lastSeenTs>  ·  Update trigger: per message (O(1)) + periodic sweep (10s)  ·  Hot path change: one Map.set()

Every incoming message updates a last-seen timestamp for its vessel ID. A sweep — independent of the ingestion loop — checks all tracked vessels for silence beyond a threshold, escalating severity if the vessel's last known position was inside or near a sensitive zone.

// per message — O(1), negligible cost
lastSeen.set(shipId, ts);

// every 10s, over distinct vessels — not the 50k/s stream
for (const [shipId, ts] of lastSeen) {
  if (now - ts > SILENCE_THRESHOLD_MS) {
    emitAlert("dark_fleet", shipId, lastKnownCell.get(shipId));
  }
}

Rendezvous Detection — Ship-to-Ship Transfer

reuses existing hot-path lookup
Data structure: Map<cellId, shipId[]>  ·  Update trigger: per message, using the cell ID already computed  ·  Hot path change: one Map update, no new geometry

The hexagon ID computed for every message during zone evaluation is reused to maintain a live occupancy map. If two or more distinct vessels remain within the same or adjacent hexagons beyond a defined duration, a potential rendezvous event is flagged — optionally suppressed near known ports where co-location is expected.

// same cellId already computed for zone lookup — no extra geometry
const occupants = cellOccupancy.get(cellId) || [];
if (!occupants.includes(shipId)) occupants.push(shipId);
cellOccupancy.set(cellId, occupants);

// periodic check, off hot path
if (occupants.length >= 2 && durationExceeded(cellId, MIN_DURATION)) {
  emitAlert("rendezvous", occupants, cellId);
}
